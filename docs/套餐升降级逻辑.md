# 套餐升降级逻辑

本文档详细说明套餐的升级、降级和排队机制。

## 概述

套餐系统支持优先级机制，用于处理用户兑换不同套餐时的升降级逻辑。

## 套餐优先级

每个套餐类型有一个 `priority` 字段，数值越大优先级越高：

| 套餐类型 | 优先级 | 说明 |
|----------|--------|------|
| 免费套餐 | 0 | 最低优先级，作为保底 |
| 基础套餐 | 10 | 入门级付费套餐 |
| 标准套餐 | 20 | 中级付费套餐 |
| 高级套餐 | 30 | 高级付费套餐 |
| 尊享套餐 | 40 | 最高优先级 |

## 升降级规则

### 兄换高优先级套餐（升级）

当用户兑换比当前套餐优先级更高的套餐时：

1. 立即激活新套餐
2. 当前套餐失效（不退款、不保留）
3. 新套餐的配额立即生效

```
当前: 标准套餐 (priority=20)
兑换: 尊享套餐 (priority=40)

结果:
- 标准套餐立即失效
- 尊享套餐立即激活
- 配额按尊享套餐规则重新初始化
```

### 兑换同优先级套餐（续费/叠加）

当用户兑换与当前套餐优先级相同的套餐时：

1. 时长叠加（延长到期时间）
2. 配额不重置（继续累计）

```
当前: 标准套餐 (priority=20), 到期: 2026-05-01
兑换: 标准套餐 (priority=20), 30天

结果:
- 到期时间延长至 2026-05-31
- 配额继续累计，不重置
```

### 兑换低优先级套餐（排队）

当用户兑换比当前套餐优先级更低的套餐时：

1. 新套餐进入排队状态
2. 当前套餐继续使用
3. 当前套餐到期后自动激活排队套餐

```
当前: 尊享套餐 (priority=40), 到期: 2026-05-01
兑换: 标准套餐 (priority=20), 30天

结果:
- 尊享套餐继续使用至 2026-05-01
- 标准套餐进入排队状态
- 2026-05-01 后自动激活标准套餐
```

## 免费套餐机制

### 免费套餐特点

1. **永久有效**：`expire_time = NULL`
2. **最低优先级**：`priority = 0`
3. **保底机制**：付费套餐到期后自动降级为免费套餐

### 免费套餐发放

用户注册时自动发放免费套餐：

```go
// 用户注册成功后
err := planService.GrantFreeTier(memberId)
```

### 降级到免费套餐

付费套餐到期后，自动降级：

```go
// 定时任务检查过期套餐
expiredPlans := planService.GetExpiredPlans()
for _, plan := range expiredPlans {
    // 检查是否有排队套餐
    queued := planService.GetQueuedPlans(plan.MemberId)
    if len(queued) > 0 {
        // 激活排队套餐
        planService.ActivateQueuedPlan(plan.MemberId)
    } else {
        // 降级为免费套餐
        planService.DowngradeToFreeTier(plan.MemberId)
    }
}
```

## 配额处理

### 升级时配额处理

升级时，配额按新套餐规则重新初始化：

```go
// 1. 清除旧配额
quotaService.ClearQuotas(memberId, oldPlanId)

// 2. 初始化新配额
quotaRules := planService.GetQuotaRules(newPlanId)
quotaService.InitQuotas(memberId, quotaRules, expireTime, newPlanId)
```

### 降级时配额处理

降级时，配额按新套餐规则重新初始化：

```go
// 1. 清除旧配额
quotaService.ClearQuotas(memberId, oldPlanId)

// 2. 初始化免费套餐配额
freeTierQuotaRules := planService.GetFreeTierQuotaRules(projectId)
quotaService.InitQuotas(memberId, freeTierQuotaRules, nil, freeTierId)
```

### 续费时配额处理

续费（同优先级）时，配额不重置，仅延长到期时间：

```go
// 仅更新到期时间
memberPlan.ExpireTime = newExpireTime
memberPlan.Update()
```

## 排队机制

### 排队套餐存储

排队套餐存储在 `member_plans` 表中，状态为 `queued`：

```sql
INSERT INTO member_plans (member_id, plan_id, status, expire_time, activated_at)
VALUES (?, ?, 'queued', ?, NULL);
```

### 排队套餐激活

当前套餐到期后，检查并激活排队套餐：

```go
// 按优先级降序获取排队套餐
queuedPlans := GetQueuedPlans(memberId) // ORDER BY priority DESC

if len(queuedPlans) > 0 {
    // 激活最高优先级的排队套餐
    ActivateQueuedPlan(memberId, queuedPlans[0])
}
```

### 取消排队

用户可以取消排队中的套餐：

```go
err := planService.CancelQueuedPlan(memberId, planId)
```

## 流程图

### 兑换流程

```
用户兑换套餐
    │
    ▼
获取当前套餐
    │
    ▼
比较优先级
    │
    ├─ 新套餐优先级更高 ──▶ 升级：立即激活新套餐
    │
    ├─ 优先级相同 ──▶ 续费：时长叠加
    │
    └─ 新套餐优先级更低 ──▶ 排队：加入排队列表
```

### 到期检查流程

```
定时任务（每小时）
    │
    ▼
查询过期套餐
    │
    ▼
对每个过期套餐
    │
    ├─ 有排队套餐 ──▶ 激活排队套餐
    │
    └─ 无排队套餐 ──▶ 降级为免费套餐
```

## 示例场景

### 场景一：新用户注册

```
1. 用户注册成功
2. 自动发放免费套餐（永久有效）
3. 配额按免费套餐规则初始化
```

### 场景二：免费用户购买标准套餐

```
1. 当前：免费套餐 (priority=0)
2. 购买：标准套餐 (priority=20)
3. 结果：升级为标准套餐，配额重新初始化
```

### 场景三：标准用户购买高级套餐

```
1. 当前：标准套餐 (priority=20), 到期: 2026-05-01
2. 购买：高级套餐 (priority=30), 30天
3. 结果：升级为高级套餐，到期: 2026-05-27
```

### 场景四：高级用户购买标准套餐

```
1. 当前：高级套餐 (priority=30), 到期: 2026-05-01
2. 购买：标准套餐 (priority=20), 30天
3. 结果：标准套餐进入排队，2026-05-01 后激活
```

### 场景五：标准用户续费标准套餐

```
1. 当前：标准套餐 (priority=20), 到期: 2026-05-01
2. 购买：标准套餐 (priority=20), 30天
3. 结果：到期延长至 2026-05-31，配额不重置
```

### 场景六：高级套餐到期

```
1. 当前：高级套餐 (priority=30), 到期: 2026-05-01
2. 到期检查：
   - 无排队套餐
   - 降级为免费套餐
3. 结果：配额按免费套餐规则重新初始化
```

### 场景七：高级套餐到期，有排队套餐

```
1. 当前：高级套餐 (priority=30), 到期: 2026-05-01
2. 排队：标准套餐 (priority=20)
3. 到期检查：
   - 有排队套餐
   - 激活标准套餐
4. 结果：配额按标准套餐规则重新初始化
```

## API 接口

| 接口 | 说明 |
|------|------|
| plan.current | 获取当前套餐 |
| plan.queued | 获取排队套餐列表 |
| plan.list | 获取项目可用套餐列表 |

## 注意事项

1. **优先级设计**：建议使用 10 的倍数，便于后续插入新等级
2. **免费套餐唯一**：每个项目只能有一个免费套餐
3. **永久套餐**：`days=0` 表示永久有效，`expire_time=NULL`
4. **配额重置**：升级/降级时配额重新初始化，续费时不重置
5. **排队顺序**：多个排队套餐按优先级降序激活
